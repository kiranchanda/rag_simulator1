Understand RAG step by step

RAG means Retrieval-Augmented Generation. This simulator shows how documents become chunks, chunks become vectors, vectors are searched, and the LLM generates an answer using retrieved context.

Step 1 — Create the knowledge base

RAG starts by collecting trusted source content. The LLM will later use this content as external memory.

1
Sources

Documents, PDFs, emails, chats

Raw content is gathered from systems your business trusts.

Goal

Build searchable knowledge

The source material should be clean, current, and traceable.

Step 2 — Split source content into chunks

Large documents are split into smaller passages so retrieval can find the most relevant pieces.

2

Step 3 — Convert chunks into vector embeddings

An embedding is a numeric representation of meaning. Similar concepts should have similar vectors.

3

Step 4 — Store embeddings in a vector database

The vector database stores each chunk, its vector, and metadata like source name, chunk ID, and tags.

4
Vector DB Records

0 records stored

Each record connects a vector embedding back to the original source chunk.

Metadata

source, chunk_id, text, vector

Metadata helps cite sources and filter results later.

Important

RAG stores knowledge outside the model

The LLM is not retrained every time you add documents. The app retrieves relevant context from the vector database at question time.

Step 5 — Ask a question and retrieve relevant chunks

The user question is also embedded. Then the system searches for chunks with similar meaning.

5

Step 6 — Generate an answer using retrieved context

The LLM receives the user question plus the retrieved chunks, then produces a grounded answer with citations.

6
Production tip: A good RAG app should show source references, confidence, and an escalation path when retrieval quality is low.
